## 人工智能岗位面试指南

本指南基于 AI Engineering Interview Process 演讲内容，总结典型流程、题型分布以及准备建议，帮助你系统性准备人工智能工程岗位面试。

## 一、整体面试流程概览

- **常见流程**
  - Recruiter / Talent Screen：15–30 分钟，确认背景、动机、薪资期望
  - 技术面试：现场编码、系统设计或代码评审
  - Hiring Manager 面试：45–60 分钟项目深挖与角色契合度
  - 行为面试：价值观与文化匹配
  - Take‑home 作业：常见时长 2–3 小时
  - Panel 面试：多位面试官联合
  - CEO / Founder 面试：15–30 分钟，通常是最终轮

- **公司示例（Doctolib Senior AI Engineer）**
  - Recruiter 面试
  - Feature building 面试
  - AI System Design 面试
  - Behavioral 面试
  - 背调与 Offer

- **节奏与周期**
  - 中位流程约 4 轮，整体 2–6 周
  - 并行面试多家公司，有助于把握节奏与谈判空间

## 二、面试题型总览

面试问题大致分为六类：

1. Theory – 理论与实践理解：LLM、RAG、Agents、评估、成本与延迟
2. Coding – 编程题：实现题与算法题
3. Project Deep Dive – 项目深挖：端到端项目与技术难点
4. AI System Design – AI 系统设计：应用架构与权衡
5. Behavioral – 行为与价值观：冲突、失败、领导力、动机
6. Home Assignments – Take‑home 作业（通常单独一轮或后续活动）

后续小节针对前五类进行展开。

## 三、Theory：理论与实践问题

- **形式**
  - 45–60 分钟，以对话为主
  - 很少单独成轮，通常与其他环节结合
  - 不要求白板或现场编码，重点在解释与推理

- **典型主题**
  - LLM 实践：temperature、上下文窗口、memory 等
  - RAG 系统：整体流程、检索方式选择、常见失败点与扩展
  - Agents 与 Tool Use：组件、工具选择、终止条件、安全性
  - 测试与评估：指标、golden dataset、幻觉检测
  - Monitoring：线上指标、漂移监测、模型回归测试
  - 成本与延迟：token 成本、模型分层、缓存策略
  - 安全与 Guardrails：prompt injection、隐私与合规、策略检测

- **示例问题**
  - 请完整解释一个 RAG 系统的流程。
  - 文本检索 vs 向量检索，各适用于什么场景？
  - 如何处理超大的 PDF 报告文档？
  - 如果检索不到相关上下文，你如何降低 LLM 幻觉？
  - RAG 系统常见失败点有哪些？你如何 Debug？
  - 如何将一个 RAG 系统扩展到处理千万级文档？
  - 什么样的系统可以称为 agentic？什么时候不应该用 Agent？
  - 如何检测并终止 Agent 的无限规划循环？
  - 你如何设计评估框架（指标、数据集、流程）？

- **准备建议**
  - 以实践为主，而不是死记硬背理论。
  - 至少独立从零搭建一个 RAG 管线和一个简单 Agent 系统。
  - 熟悉如何构建 golden dataset 与评估 pipeline。
  - 回答时关注 trade‑off：不仅说明 “怎么做”，还要说明 “什么时候这样做，为什么不用其他方案”。

## 四、Coding：编程与算法

- **两种常见形式**
  - Implementation（实现题）
    - 时长：45–90 分钟
    - 形式：一题多阶，逐步增加复杂度
    - 重点：代码质量、扩展性、工程实践
  - Algorithm（算法题）
    - 时长：25–70 分钟
    - 形式：接近 LeetCode
    - 重点：数据结构与算法思维

- **实现题示例**
  - 构建一个网站爬虫
  - 重构 100–120 行结构混乱的代码
  - 实现一个 KV 数据库：`SET` / `GET` / `DELETE`
  - 实现 Unix `cd` 命令，处理符号链接
  - 实现一个内存数据库，支持类 SQL 操作
  - 实现带复杂规则的积分 / Credits 管理系统

- **算法题示例**
  - RLE 编码：`aaabbcca` → `a3b2c2a1`
  - 0 到 100 间所有质数
  - O(1) 时间复杂度的 LRU Cache
  - 反转链表（可能是 AI‑assisted 轮次）
  - 根据列号输出 Excel 列名

- **准备建议**
  - 至少刷 75+ 道简单/中等难度 LeetCode。
  - 重点掌握：数组与列表、集合、哈希表、Trie、链表等基础结构。
  - 做有层次的项目练习，习惯应对“在现有实现上增加新需求”。
  - 练习“边写边讲”：持续解释思路、权衡与下一步计划。
  - 熟悉在允许使用 AI 工具时，如何与之协作而不是盲目粘贴答案。

- **常见失误**
  - 没有先澄清需求与边界条件就直接写代码。
  - 写出无法扩展的硬编码方案，一改需求就整体重写。
  - 不理解 AI 生成的代码，无法在面试官追问时解释清楚。

## 五、Project Deep Dive：项目深挖

- **形式**
  - 30–60 分钟，以对话为主
  - 常见模式：自由对话型深挖
  - 较少见：事先准备好的正式演示

- **典型问题**
  - 请从头到尾介绍一个你亲自主导或参与的 AI 项目。
  - 请讲一个你认为技术上最有挑战的项目。
  - 最近一次或最喜欢的项目是什么？遇到哪些难点？
  - 描述一个你解决复杂 prompt engineering 问题的经历。
  - 说一个你最自豪的项目，以及原因。

- **追问维度（面试官会重点深挖）**
  - 业务背景：解决什么问题？为什么重要？你的角色是什么？
  - 决策过程：为何选择当前方案？和替代方案的权衡？
  - 问题与 Debug：遇到什么失败？线上问题如何排查与修复？
  - 结果与评估：是否真正有效？用什么方式衡量？
  - 反思与迭代：如果重来一次，你会做哪些不同？下一步想改进什么？

- **准备建议**
  - 准备 2–3 个可以讲 15–20 分钟的代表性项目。
  - 对每个项目，从“问题–方案–实现–评估–上线–迭代”全链路梳理清楚。
  - 尤其强调“为什么这么做”和“如何评估效果”，而不仅是技术堆栈列表。

## 六、AI System Design：AI 系统设计

- **形式与结构**
  - 时长通常 45–60 分钟。
  - 常见时间分配：
    - 需求澄清：约 5 分钟
    - 高层架构：10–15 分钟
    - 组件深挖：15–20 分钟
    - 权衡与瓶颈：5–10 分钟
    - 后续场景与扩展：剩余时间

- **典型问题**
  - 设计一个聊天机器人（类似 ChatGPT、Claude）。
  - 设计一个文档问答 / RAG 系统。
  - 设计一个 AI Copilot（类似 GitHub Copilot）。
  - 设计一个医院语音助手。
  - 设计 ChatGPT 的跨会话记忆系统。
  - 设计一个多步 agentic 工作流。
  - 设计一个类似 Perplexity 的实时 LLM 搜索引擎。

- **与传统系统设计的差异**
  - 数据：更注重上下文工程与文本切分，而不仅是特征工程与训练数据。
  - 输出：开放式文本或代码，而非简单分数或标签。
  - 确定性：结果非确定性，需要更复杂的评估与监控。
  - 评估方式：LLM‑as‑judge、人评、任务成功率等，而不仅是精准率/召回率。
  - 成本结构：按 token 持续计费，而非仅训练期计算资源。
  - 失败模式：幻觉、注入攻击等，而不是传统的数据漂移、偏差。

- **准备建议**
  - 使用稳定的五步结构回答：
    1. 问题定义：用户、场景、约束、Guardrails
    2. 高层架构：关键组件与数据流
    3. 组件深挖：RAG、工具调用、记忆、评估、安全等
    4. 权衡与故障：会在哪些点出问题？如何监控与降级？
    5. 收尾总结：重申方案、主要风险与下一步迭代方向
  - 熟悉可复用的模式：RAG 模式、反馈闭环、幻觉缓解、可扩展性与成本优化。
  - 回答时避免只堆叠框架/产品名，要解释为什么选它们以及替代方案。

- **常见失误**
  - 不先澄清需求，直接画架构图。
  - 把 LLM 当“真理源”，而不是检索/工具之上的生成层。
  - 只讲 happy path，不考虑失败场景、监控指标与回滚方案。
  - 只谈“我会用某某框架”，不解释取舍与实现细节。
  - 忽略成本与延迟预算，也没有缓存与路由策略。
  - 完全忽略安全与隐私问题。
  - 为了“酷”而强行上 Agent，而不是从问题出发。

## 七、Behavioral：行为与价值观

- **形式**
  - 30–60 分钟
  - 典型开头：“Tell me about a time you…”

- **问题类别示例**
  - AI 相关：你做过哪些与 AI 相关的 side project？
  - 冲突：讲一个你和他人产生具体冲突的案例。
  - 领导力：讲一个你主动承担责任或主导项目的经历。
  - 决策：一次错误技术决策导致延期或问题的经历。
  - 失败：一个明确的失败案例，以及学到什么。
  - 动机：为什么想加入这家公司？为什么现在考虑换工作？

- **准备建议**
  - 找到目标公司的价值观，每条价值观准备 2–3 个故事。
  - 使用 STAR 结构讲述：Situation / Task / Action / Result。
  - 如果公司没有公开价值观，可以借用 Amazon Leadership Principles 做参考。
  - 故事要具体、可验证，避免过度包装。

## 八、核心技能与差异化

- **基础技能（几乎所有公司都会期望）**
  - RAG 设计与实现：能够回答设计题和实现细节。
  - 向量数据库：理解 Pinecone、FAISS、Qdrant 等的取舍。
  - 工具调用与 Agent：函数调用、Agent 架构与系统设计。
  - Python：编程轮与项目实现的主要语言之一。
  - API 与部署：如何把模型能力暴露为 API、集成到产品中。

- **可以拉开差距的能力**
  - 评估框架：能自己设计并实现评估 pipeline。
  - 成本与延迟意识：用前后对比的方式讲成本优化故事。
  - 可观测性：日志、指标、Tracing、在线 A/B 等。
  - 系统图与文档：能画清楚架构并用文档解释。
  - 真正的生产经验：上线、监控、故障处理的完整闭环。
  - AI Fluency：自然、自信地讨论模型、产品与业务影响。

- **优秀候选人的特征**
  - 前 5 分钟就能清晰展示影响力与亮点，而不是堆技术名词。
  - 说话像“Builder”而不是“纯研究者”，注重取舍与工程实现。
  - 对成本敏感，能讲清楚“上线前/后”的成本变化。
  - 诚实面对能力边界，坦诚说“不知道”，并说明如何补齐。
  - 不需要是“全能独角兽”，在 1–2 个方向有明显深度即可。

## 九、面试后的策略

- **面对拒信**
  - 尽量不要情绪化地解读结果。
  - 做一次复盘：哪些环节发挥不好？在哪些问题上暴露了能力短板？
  - 很多拒信原因与你个人无关：例如有人更匹配、预算变化、优先级调整等。
  - 如果确实发现技能差距，把它转化为一个新的项目去补齐。

- **面对 Offer 与谈判**
  - 不要立刻口头答应，留出时间评估。
  - 如果还有其他面试流程在走，可以先完成，形成多个选择。
  - 可以在合适范围内告知其他公司你已有 Offer，以加快他们进度。
  - 多个 Offer 是谈判中的理想状态，更容易争取到合适的薪资与 title。

## 十、备战建议小结

- **项目与作品集**
  - 两三个端到端项目（RAG、Agent、真实部署）往往比一堆证书更有说服力。
  - 在项目中展示“从原型到省钱高效的生产系统”的演进过程。
  - 打造干净的 GitHub：清晰的 README、演示、文档与评估脚本。

- **评估与工程化**
  - 尽早学习评估工具与框架：如 Ragas、DeepEval、LLM‑as‑judge 等。
  - 展示工程成熟度：Docker、CI/CD、监控告警等能力。
  - 理解并能解释成本与延迟优化：缓存、批处理、模型路由等。

- **表达与叙事**
  - 多练习“讲故事”，而不是背标准答案。
  - 面试中优先强调“业务影响与结果”，再展开使用的技术。
  - 在理论、Coding、项目、系统设计与行为面试中，始终围绕“影响力 + 取舍 + 评估”这三条主线来回答问题。
