## 人工智能工程岗位要求概览

本页基于 2026 年 1 月从 builtin.com 收集的 **895 条 AI Engineer 岗位 JD（来自 590 家公司）** 的结构化分析，总结当前市场对「AI 工程师」的真实要求，包括角色类型、技能栈、日常职责和典型业务场景。

- **岗位类型**
  - **AI-First（69.4%）**：直接构建 RAG、智能体、LLM 功能，是产品核心能力的一部分。
  - **AI-Support（28.5%）**：为其他团队提供平台、基础设施、工具（MLOps、推理平台、内部框架等）。
  - **ML 型（1.8%）**：传统 ML 岗位的改名，侧重经典建模。
  - **Applied vs Research**：**95.6% 为应用/生产导向，仅 4.4% 偏研究**。市场更偏好「能上线产品」而不是「发论文」的人。

## 技能要求：AI 工程师是 Full-Stack Builder

- **整体画像**
  - **93.1% 岗位要求 GenAI 以外的技能**，AI 工程师本质是能把 AI 融入产品的全栈工程师，而不是只会调 API。
  - 前端技能：约 31% 岗位提到；后端技能：约 50%；前后端都要：约 22%。
  - **仅 ~1.4% 岗位期望「纯 GenAI 工作」**（只做模型/提示工程，不管工程落地）。

- **核心技术栈（Top Skills）**
  - **语言与编程**
    - **Python（82.5%）**：事实上的必备语言，用于服务端、数据处理、RAG/Agent、评估等。
    - TypeScript（23.4%）、Java（14.9%）、Go（11.3%）、SQL（9.8%）：补充后端/系统开发能力。
  - **云与基础设施**
    - **AWS（40.1%）**、Azure（23.9%）、GCP（22.9%）：至少熟悉一个主流云平台。
    - **Docker（31.0%）**、**Kubernetes（29.1%）**：容器化、编排、部署是基本要求。
    - CI/CD（29.3%）、Terraform、MLOps：有一整套「可持续交付与运维」能力更有竞争力。
  - **GenAI 与框架**
    - RAG（35.9%）：最常见的技术模式。
    - Prompt engineering（29.1%）、LLM 使用（25.4%）。
    - 框架生态：LangChain（18.8%，占主导）、LangGraph（8%）、LlamaIndex（5.8%）、CrewAI、AutoGen 等。
    - API：OpenAI、Anthropic 等主流提供商。
  - **数据库与存储**
    - 各类 **向量数据库**（general 97 条）、Pinecone、Weaviate、Redis 等。
    - **PostgreSQL 依然是关键通用数据库**，配合 pgvector 等扩展支持向量搜索。

- **ML 能力的深度**
  - 约 **64% 的 AI-First 岗位要求「一定的 ML 知识」**，但重点是 **实战而非研究**：
    - PyTorch（26.6%）、Fine-tuning（25.6%）、TensorFlow、Embeddings、模型评估等。
  - **微调（fine-tuning）被认为「被过度吹捧」**：
    - 只有 4% 岗位将其作为主要职责，15% 偶尔用，80% 完全不提。
    - 学习重点应先放在 **RAG 与 Agent**，再根据垂直行业（医疗、金融、法律等）补充微调技能。

## 典型职责：从构建到上线和评估

大规模分析 5,694 条「Responsibilities」后，可以把 AI 工程师的日常工作概括为几大类：

- **构建 AI 系统（非常常见）**
  - 设计与开发 RAG 系统（检索、语义搜索、重排等）。
  - 构建基于 LangChain、LangGraph、CrewAI 等的智能体与多步工作流。
  - 实现 LLM 驱动的具体产品功能（聊天、助手、搜索、自动化工具等）。
  - 建立 Prompt/模板库和评估/测试框架。
  - 关键能力：**把业务问题转化为可维护、可扩展的 AI 系统**。

- **生产化与运维（非常常见）**
  - 以 API/微服务形式交付 AI 功能，接入测试、QA、监控和告警体系。
  - 参与从需求到上线全流程，对功能质量和稳定性负责，必要时轮值 On-call。
  - 构建推理与服务基础设施（包括 GPU 集群、负载均衡、多区域部署等）。
  - 核心挑战：**让不确定性的 AI 系统在生产环境足够可靠**。

- **评估与质量（非常常见）**
  - 搭建评估框架与测试工具，定义合适的指标。
  - 使用 LLM-as-judge、黄金数据集等方法做离线评估与回归测试。
  - 监控输出质量、安全性、偏见、漂移与幻觉等问题。
  - 市场上约 **39.6% 的 AI-First 岗位显式要求「评估能力」**。
  - 关键信息：**会「测」比只会「做」更具差异化**。

- **使用第三方 LLM 提供商（非常常见）**
  - 集成 OpenAI、Anthropic 等 API，处理错误、重试、回退策略。
  - 优化提示设计和调用策略，以满足不同模型、不同任务的需求。
  - 通过缓存、配额管理、降级方案等控制成本与速率限制。

- **RAG 与检索相关工作（常见）**
  - 面向文档、网页、音视频等多源数据构建检索和索引管道。
  - 实现语义检索、混合检索、查询重写、重排序等机制。
  - 管理长上下文、多轮对话中的上下文注入与窗口控制。

- **数据处理与数据工程（常见）**
  - 建立数据管道完成数据收集、清洗、转换、索引与版本管理。
  - 与内部数据仓库/数据湖打通，为 RAG、评估和监控提供稳定数据源。

- **协作与沟通（常见）**
  - 与产品、设计、业务团队一起定义路线图和需求。
  - 在技术与非技术团队之间翻译和沟通 AI 能力与限制。
  - 指导/辅导其他工程师，并撰写文档、维护知识库。

- **基础设施与平台（常见，偏 AI-Support 岗）**
  - 设计和构建可扩展的分布式 AI 平台，供公司内部工程师使用。
  - 搭建 GPU 调度、模型注册、监控、配额与安全合规体系。

总体来看，岗位描述中最常见的动词是 **build / deploy / design / collaborate / implement / develop**，强调的是「**动手打造与落地**」而非理论研究。

## 典型业务场景与用例

对 4,525 条 use cases 的分析显示，AI 工程师主要服务于以下几类业务问题：

- **自动化人工工作流（约 15.4%）**
  - 如跨 Salesforce 的业务流程自动化、IT 运维自动化、供应链编排等。
  - 解决方案多是具备记忆与工具调用能力的 **多步骤智能体**。

- **提升内部运营效率（约 11.5%）**
  - 风险与威胁早期预警、保险理赔自动化审核、金融风控、合规审查等。
  - 重点是安全、可审计、可控的企业级 AI 系统。

- **在企业数据中查找信息（约 8.0%）**
  - 大规模企业知识检索、医学文献搜索、金融信息查询、基于知识图谱的 RAG 等。
  - 典型方案是结合向量检索、混合检索和图结构的 **语义搜索与问答系统**。

- **大规模回答客户问题（约 6.9%）**
  - 客服自动化、语音/多模态客服、医疗随访对话、用户支持机器人等。
  - 核心是 **带公司知识的客户交互代理**。

- **可靠部署与推理服务（约 4.8%）**
  - 低延迟推理、边缘推理、全球多地区部署等。
  - 主要挑战在于延迟、吞吐、稳定性与成本控制。

- **数据驱动决策（约 3–4%）**
  - 金融、医疗、营销等领域的数据分析与预测辅助决策。

- **AI 质量与安全（约 3%）**
  - 内容安全检测、隐私保护、安全合规、模型输出审核等。

- **其他场景**
  - 个性化推荐、内容生成、开发者辅助写代码、处理高度专业领域知识等。
  - 金融、医疗、教育、安全、法律监管、制造、零售等行业需求尤为集中。

## 能力与学习路径建议

- **关键结论**
  - **大多数岗位是 AI-First**：主打 RAG、智能体和 LLM 应用。
  - **AI 工程师是 Builder**：强调构建、部署、监控、评估完整系统。
  - **Python 几乎是硬性要求**，其次是云平台 + Docker/K8s 等生产工程能力。
  - **RAG 是最重要的技术模式，自动化工作流是最重要的业务用例**。
  - **评估与监控将越来越成为核心竞争力**。

- **推荐学习路径（核心）**
  1. **基础**：理解 LLM 工作方式、RAG 与搜索基础。
  2. **RAG 实战**：围绕 FAQ、文档问答、多数据源搭建多个 RAG 项目。
  3. **智能体**：学习函数调用、工具使用、代理框架（如 LangChain、LangGraph 等）。
  4. **测试与成本**：为 Agent/LLM 构建测试，实践 LLM-as-judge、成本监控。
  5. **监控与可观测性**：日志、Tracing、OpenTelemetry、Grafana 等。
  6. **评估**：离线评估、合成数据、提示优化。
  7. **生产化**：部署、权限与安全、Guardrails，以及端到端项目落地。

- **技能优先级**
  - **必须具备**：Python、Prompt Engineering、RAG 模式理解、至少一个云平台、Docker。
  - **高价值补充**：智能体框架（LangChain/LangGraph 等）、TypeScript、FastAPI、Kubernetes、CI/CD。

- **转岗难度与优势（从其他岗位出发）**
  - **ML Engineer → AI Engineer**：难度最低，核心变化是「自训练模型」变为「调用 API + 系统集成」。
  - **Data Engineer / Backend Engineer**：2–4 个月可平滑过渡，数据管道/后端服务经验对 RAG 和生产化非常有用。
  - **Data Scientist**：评估和实验设计是天然优势，需要补足工程与系统构建能力。
  - **Frontend Engineer**：路径更长，但具备端到端产品思维和前端实现能力，在 AI 产品团队中很吃香。

整体来说，「人工智能工程岗位」的本质是：**在真实业务环境中，用 LLM + RAG + Agents 等能力构建、上线并持续维护可靠的 AI 系统**，而不仅仅是做 Demo 或实验。
